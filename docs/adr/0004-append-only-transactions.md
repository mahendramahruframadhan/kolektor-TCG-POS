# ADR-0004: Append-only transactions enforced at the SQL level

**Status:** Accepted · 2026-04-24

## Context

KolektaPOS runs real money: sales, voids, refunds, per-owner settlements. A mutation pattern where a row can be `UPDATE`-d or `DELETE`-d opens a class of bugs (or abuse) that is very hard to audit after the fact. We need a tamper-resistant financial record.

## Decision

`transactions`, `transaction_items`, and `audit_log` are **append-only**:

1. **DB triggers** in `packages/db/src/triggers.sql` raise `SQLITE_CONSTRAINT` on any `UPDATE` or `DELETE` against these three tables.
2. **ORM layer** (route handlers) never emits `db.update(transactions)` or `db.delete(...)`. Enforced by code review + `grep` on PR.
3. **Void / refund** is a *new* row with `kind = 'void' | 'refund'` and `parent_transaction_id` set. Totals are stored negative so reports compute net by summing.
4. **Reports** compute per-owner payout by summing `transaction_items.sold_price_idr` (already signed, no kind-based multiplier — see settlement bugfix in Phase 3).

## Consequences

- Any attempt to silently edit a paid transaction crashes hard at the DB layer, not at the app layer.
- Reporting is deterministic: net = `SUM(sold_price_idr)`. Grand totals use `SUM(sale.total_idr) - SUM(ABS(void.total_idr))`.
- Schema migrations that modify these tables need careful handling (drop trigger → alter → re-create trigger).
- Triggers are hand-authored SQL — not generated by drizzle-kit. See `packages/db/src/migrate.ts` for the post-migration `sqlite.exec(triggers)` step.

## Alternatives considered

- **App-level append-only without DB triggers.** Rejected — one stray `db.update(transactions)` in a PR breaks the invariant silently.
- **Soft-delete column (`deleted_at`).** Rejected — voids are not deletes; they're new financial events with their own timestamps.
