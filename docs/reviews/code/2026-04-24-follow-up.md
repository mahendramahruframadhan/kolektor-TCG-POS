## Executive Summary

KolektaPOS has moved from "MVP-complete but rough around operational edges" to "shippable for a controlled first event" in a single sprint. Six improvement commits closed the long-standing gaps in compliance (LICENSE + retention policy + audit-log pruner), documentation (README rewrite + CONTRIBUTING + six ADRs), observability (`/health` + structured pino events + client-side sync-error surfacing), performance (pagination, SQL-pushed monthly filter, xlsx code-split → 30% smaller main bundle), and configuration (fail-fast Zod env validation, per-key settings schemas, CORS dev allowlist). Accessibility shipped earlier still holds — WCAG AA form labels + Dialog primitive + keyboard tap-and-hold + palette contrast fixes are all in place. **60 tests across 14 files pass**, `pnpm typecheck` + `pnpm build` are green, and the branch is ready for merge once three known business-logic bugs are closed.

The three that block a confident ship: the **oversold-void handler still flips `cards.status='available'` even when another sale on the same card is still open** (H-1); **settled events are not locked** against late-arriving transactions (H-2); and the **cashier bootstrap still calls the admin-gated `/users` endpoint** (H-3), so a fresh cashier device cannot hydrate. None of these were in the improvement scope and none regressed — they're carryover from the earlier merged review, now the top unresolved items.

### Top 5 risks
- **High / High / S — Oversold void reopens the card.** `apps/api/src/routes/transactions.ts:200-208` unconditionally sets `cards.status = 'available'` on void, even when a second un-voided sale exists on the same card. Voiding half of an oversold event makes the card re-sellable a third time.
- **High / High / S — Settled events are not locked.** No `cart.event.settledAt` check in any mutating route. A late sync push can re-open payouts after admin distributes cash.
- **High / High / S — Cashier bootstrap calls admin-only `/users`.** `apps/web/src/lib/sync.ts:22` still invokes `api.users.list()` which is gated by `requireAdmin`. Non-admin sync-pull fails at first-device hydration.
- **High / High / M — `/sync/push` handles only `create_card` + `create_transaction`.** Offline-queued cart/item/hold ops strand indefinitely (`unsupported_op_type`). The PWA is therefore not truly offline-first for cashier writes (PRD invariant #1).
- **Medium / High / M — `/sync/pull` cursor=0 still unpaginated.** Initial hydration loads every user, every event, every card, every last-30-day transaction + its items in one response. Fine for 11 users on day one; scales linearly and will hurt as data accumulates.

### Quick wins (S effort)
- Gate the void-status-reset on "no remaining unvoided sale for this card" (see H-1 fix sketch).
- Reject cart/pay/void where `event.settledAt !== null`.
- Split cashier bootstrap onto a dedicated `/auth/bootstrap` route returning the minimal user DTO without needing `requireAdmin`.
- Move the remaining two `useId()` form-label migrations (StockReceivePage + PaymentModal inner labels) into a single commit.
- Honour the "active + last 2 closed" event filter on `/sync/pull` cursor=0 — the comment already describes the intended behaviour.
- Remove the committed `.env` from operator disk; rotate `SESSION_SECRET` before the first event (`loadConfig()` now rejects the placeholder, so anyone running with the default will fail-fast at boot).

## Architecture Snapshot

**Monorepo** (Turbo + pnpm 10, Node 22): `apps/{web,api}` + `packages/{db,types,sync,qr,ui}`.

- **API** (`apps/api`): Fastify 5 + better-sqlite3 11 via Drizzle 0.38. Plugins (in register order): `helmet`, `cors` (dev allowlist or `https://${DOMAIN}`), `rate-limit`, `swagger` + `swagger-ui` (`/docs/api`), custom `sessionPlugin`, custom `auditPlugin`. Background jobs: `startCartSweeper` + `startAuditPruner`. Env validated by `loadConfig()` at boot.
- **Web** (`apps/web`): React 19 + Vite 6 + vite-plugin-pwa. Dexie 4 IndexedDB. TanStack Query 5 (localStorage persister), Zustand 5 UI state + new `sync-state` store. `xlsx` dynamic-imported to keep the main chunk light.
- **Packages**: `db` (Drizzle schema + SQL triggers + migration loader), `types` (Zod schemas shared monorepo-wide, now with `SETTING_SCHEMAS`), `sync` (protocol + conflict resolution), `qr` (short-ID generator), `ui` (empty placeholder).

**Key data flows:**
1. **Stock-receive:** client generates short-ID → `POST /cards` (Zod-validated, session-sourced `stockReceivedByUserId`) → stored in SQLite → Dexie.
2. **Checkout:** scan → Dexie lookup → optimistic lock + `POST /carts/:id/items` (fixed-price floor + negotiable-bottom floor + discount-pct + admin-override role check) → `POST /carts/:id/pay` (inserts append-only transaction + items with `ownerUserIdSnapshot`, emits `sale_completed` + `oversold_detected` pino events, releases locks).
3. **Sync pull:** cursor=0 returns every entity (unbounded — known gap); cursor>0 returns deltas for `cards`/`events`/`users`/`carts`/`transactions` only (five tables still missing from delta branch).
4. **Sync push:** `SyncPushRequestSchema` envelope + per-op `.strict()` schemas for `create_card` + `create_transaction` only; all others return `unsupported_op_type`.
5. **Void/refund (admin-only):** new append-only transaction with negative totals + mirrored negative items; resets `cards.status='available'` without checking for other unvoided sales (H-1).
6. **Settlement:** reads `transaction_items.ownerUserIdSnapshot`; per-owner aggregator sums signed `soldPriceIdr` directly.

**Deps:** Fastify 5.2, better-sqlite3 11.7, Drizzle 0.38, bcryptjs 3, `@fastify/{helmet, cors, rate-limit, session, cookie, swagger, swagger-ui}`, archiver 7, node-cron 3, zod 3.24. Web: React 19, Vite 6, Dexie 4, TanStack Query 5, Zustand 5, react-router 7, html5-qrcode 2, xlsx 0.18 (dynamic), uuid 11.

## Findings (Prioritized)

### Critical

### High

#### H-1. Void/refund reopens the card even when another sale remains

- **Severity / Confidence / Effort:** High / High / S
- **Category:** Functionality / PRD invariant #10
- **Location:** `apps/api/src/routes/transactions.ts:199-208`
- **Problem:** The void/refund handler does `db.update(cards).set({ status: 'available', … })` for every item on the parent transaction, with no check for another `sale`-kind `transaction_item` on the same card that has *not* been voided. Voiding one of two oversold sales therefore frees the card for a third sale.
- **Impact:** Inventory count drifts from ground truth after any oversold resolution.
- **Recommendation:** Before resetting status, count remaining unvoided `sale` items for each `cardId`. Keep `status='sold'` if any remain; clear the `oversold` flag only if exactly one remains.
- **Notes / Example Fix:**
  ```ts
  for (const cardId of cardIds) {
    const remaining = db.select({ c: count() })
      .from(transactionItems)
      .innerJoin(transactions, eq(transactionItems.transactionId, transactions.id))
      .where(and(
        eq(transactionItems.cardId, cardId),
        eq(transactions.kind, "sale"),
        notExists(/* matching void-child tx */)
      )).get()?.c ?? 0;
    if (remaining === 0) {
      db.update(cards).set({ status: "available", oversold: false, updatedAt: nowSec }).where(eq(cards.id, cardId)).run();
    } else if (remaining === 1) {
      db.update(cards).set({ oversold: false, updatedAt: nowSec }).where(eq(cards.id, cardId)).run();
    }
    // else: leave sold + oversold untouched
  }
  ```

#### H-2. Settled events are not locked against new transactions

- **Severity / Confidence / Effort:** High / High / S
- **Category:** Reliability
- **Location:** `apps/api/src/routes/settlement.ts` (settle handler stamps `settledAt`); `apps/api/src/routes/carts.ts` (no check on mutations); `apps/api/src/routes/transactions.ts` (no check on void/refund).
- **Problem:** After `POST /events/:id/settle` writes `settledAt`, nothing rejects subsequent cart creation, add-item, pay, void, or refund against that event. A late-arriving sync push from an offline device can therefore re-open payouts after admins have distributed cash.
- **Impact:** Post-settlement re-openings silently corrupt paid-out payouts.
- **Recommendation:** Add `requireEventNotSettled(db, eventId)` helper; apply as a preHandler-like check on every cart mutation and void/refund. For `/sync/push`, surface `settled_event_rejected` in the op result so the client sees it.

#### H-3. Cashier bootstrap calls admin-gated `/users`

- **Severity / Confidence / Effort:** High / High / S
- **Category:** Functionality / Offline-first
- **Location:** `apps/web/src/lib/sync.ts:22` → `api.users.list()`; `apps/api/src/routes/users.ts:16` guarded by `requireAdmin`.
- **Problem:** `fetchAndSync()` is called on first-device hydration and retries on reconnect. The `/users` call returns 403 for non-admin sessions, which collapses the whole `Promise.all` of the initial sync. Dashboard stats, reports, and offline functions all end up empty.
- **Impact:** New cashier devices cannot hydrate. `/sync/pull` user payload (without `passwordHash`) is already correct — only the client call path is wrong.
- **Recommendation:** Route the cashier bootstrap through `/sync/pull?cursor=0` (which already returns users via `userDto`). Remove the `api.users.list()` call in `fetchAndSync`. Keep `/users` admin-only for management.

#### H-4. `/sync/push` still missing handlers for cart/item/hold ops

- **Severity / Confidence / Effort:** High / High / M
- **Category:** Offline integrity
- **Location:** `apps/api/src/routes/sync.ts` push handler switch — only `create_card` + `create_transaction` cases. Default branch → `unsupported_op_type`.
- **Problem:** The client enum at `packages/sync/src/protocol.ts` defines `create_cart`, `update_cart`, `add_cart_item`, `remove_cart_item`, `abandon_cart`, `pay_cart`, `create_hold`, `release_hold` — none have server handlers.
- **Impact:** Any op queued offline against these types is rejected on reconnect. PWA is not truly offline-first for cashier writes despite the PRD invariant #1.
- **Recommendation:** Expand push with strict per-op payload schemas (mirror the `CreateCardPushPayloadSchema` pattern). Ensure `cashierUserId` / `heldByUserId` always come from the session. Maintain idempotency by `clientId`. This is the missing half of the original C3 finding.

#### H-5. `/sync/pull` initial dump unbounded and single-shot

- **Severity / Confidence / Effort:** High / Medium / M
- **Category:** Performance
- **Location:** `apps/api/src/routes/sync.ts` — cursor=0 branch returns every user/event/card/last-30-day-tx+items in one response. `hasMore: false` always.
- **Problem:** Pagination landed on `/cards` and `/transactions` list endpoints but the sync-pull initial dump was not touched. After a few events the payload runs into the multi-MB range.
- **Impact:** Mobile devices on convention-hall Wi-Fi may time out hydrating a new device.
- **Recommendation:** Chunk per entity type (≤ 500 rows/page) with `hasMore` + a next-page token. Filter events to "active + last 2 closed" as the existing comment already describes. Enable gzip at Fastify.

#### H-6. Delta sync still misses five entity types

- **Severity / Confidence / Effort:** High / High / S (per table)
- **Category:** Sync completeness
- **Location:** `apps/api/src/routes/sync.ts` delta branch emits cards, events, users, carts, transactions only.
- **Problem:** `settings`, `payment_channels`, `transaction_items`, `holds`, `cash_reconciliations` are only returned on cursor=0.
- **Impact:** Admin-changed settings never propagate mid-event; payment-channel renames are invisible; client-computed reports under-count transaction_items arriving after cursor=0.
- **Recommendation:** Add the five tables to the delta branch. `payment_channels` needs `updatedAt`/`version` columns (schema change — small migration).

- **Severity / Confidence / Effort:** Medium / Medium / S

#### M-3. `parent_transaction_id` has no foreign-key reference

- **Severity / Confidence / Effort:** Medium / High / S
- **Category:** Data integrity
- **Location:** `packages/db/src/schema.ts:227` — `parentTransactionId: text("parent_transaction_id")` with no `.references(() => transactions.id)`.

#### M-4. `.find()` after `.all()` on void existence check

- **Severity / Confidence / Effort:** Medium / Medium / S

- **Problem:** `db.select().from(transactions).where(eq(parentTransactionId, parentId)).all().find((t) => t.kind === 'void')` — JS-level scan.
- **Recommendation:** Push predicate into SQL: `…where(and(eq(parentTransactionId, parentId), eq(kind, 'void'))).get()`.

#### M-5. Background sync cursor can advance on partial apply failure

- **Severity / Confidence / Effort:** Medium / Medium / S
- **Category:** Reliability
- **Location:** `apps/web/src/lib/background-sync.ts` — `deltaSyncPull` advances `setSyncCursor(response.newCursor)` unconditionally after `applyChanges`.
- **Problem:** If a single change fails (`console.warn` in `applyChanges`), the cursor still moves past it and that change is never retried.
- **Recommendation:** Track `applyChanges` result (per-entity-type pass/fail counters); only advance the cursor when every change applied. On partial failure, leave the cursor and surface error via `useSyncStateStore` (already wired for total failure).

#### M-6. `/events`, `/users`, `/holds/active`, `/payment-channels`, `/audit-log` list endpoints unpaginated

- **Severity / Confidence / Effort:** Medium / High / S
- **Category:** Performance

- **Problem:** Pagination helper was only wired into `/cards` and `/transactions`. Remaining list endpoints still load every row.
- **Recommendation:** Import `parsePagination` and apply the same pattern. Five-file mechanical change.

#### M-7. Cart sweeper misses orphan locks

- **Severity / Confidence / Effort:** Medium / Medium / M
- **Category:** Reliability / PRD invariant #9
- **Location:** `apps/api/src/jobs/cart-sweeper.ts`
- **Problem:** The sweeper releases cards whose cart is idle *and* has a `cart_items` row. A crash between `cards.locked_by_cart_id` set and `cart_items` insert leaves orphan locks that no sweep catches.
- **Recommendation:** Additionally sweep cards with `locked_at < now − ttl*2` whose `lockedByCartId` points to a non-draft cart OR has no matching `cart_items` row.

#### M-8. POS pay-failure ambiguity (no server reconcile)

- **Severity / Confidence / Effort:** Medium / Medium / M
- **Category:** UX / Reliability
- **Location:** `apps/web/src/pages/POSPage.tsx` pay handler
- **Problem:** If `api.carts.pay` fails after server commit (network timeout on response), the client shows a generic error; the cart stays `draft` locally while the cards may already be `sold` server-side.
- **Recommendation:** Use the already-present `transactionClientId` for idempotency. On network failure, query `/transactions?clientId=…` before showing an error. Add "Check transactions" + "Retry" actions.

#### M-9. Photo upload endpoint still a stub

- **Severity / Confidence / Effort:** Medium / High / M

- **Location:** `apps/api/src/routes/sync.ts` — `POST /sync/photo/:cardClientId` updates `photoPath` without reading the multipart body.

## Detailed Review by Criteria

### 1) Functionality & Code Quality

**Strengths.** PRD invariants cited inline near enforcing code (append-only triggers, settlement snapshot, cart-lock denormalisation, fixed-price floor). `.strict()` Zod everywhere on client-to-server boundaries including `/sync/push`. Session-sourced identity (`cashierUserId`, `stockReceivedByUserId`) — client can no longer forge attribution. The recent `/simplify` pass eliminated duplicate SELECTs via request-stashed `cart`/`hold` preHandler, killed a dead `cardRows` query in `/sync/pull`, consolidated test migration loading into `applyDrizzleMigrations()`.

**Issues.** H-1 oversold-void status bug. H-2 settled-event lock missing. M-2 StockReceivePage still has visual-only labels. M-3 `parent_transaction_id` lacks FK. M-4 JS `.find()` after `.all()` for void existence.

**Recommendations.** Introduce a shared `<LabeledInput>` helper in `apps/web/src/components/` so the remaining form-label migration and PaymentModal inner fields can all be fixed in one change. Adopt a lint rule (once L-1 lands) banning raw spreads into `db.insert(...).values(...)` from untrusted input.

### 2) Testing

**Strengths.** 60 tests / 14 files across the monorepo. Integration tests spin up Fastify + in-memory SQLite with the real Drizzle migrations via `applyDrizzleMigrations()`. Covered: append-only triggers (load-bearing invariant), seed bcrypt + env-gate, sync redaction + push validation, authz boundaries, fixed-price floor, settlement sign, backup zip shape, inverted tap-hold reveal. All green.

**Issues.** No test for `/carts/:id/pay` happy-path (the highest-value business endpoint). No cart-sweeper cron test. No test for the new `/health` endpoint or the audit pruner. No oversold resolution test (would have caught H-1). No e2e browser test.

**Recommendations.**

### 3) Security

**Strengths.** bcryptjs cost 12 (ADR-0001). Session cookie 30-day rolling, HttpOnly, SameSite=Strict, Secure in production (ADR-0002). Helmet defaults + strict CORS (now explicit dev allowlist, **not** reflect-origin) + rate-limit (20/min on `/auth/login`, 10/min on `/auth/change-password`). bcrypt-only password path (`sha256:` branch removed). Seed fails fast without explicit admin vars. `/sync/pull` strips `passwordHash` via `userDto()`. `/sync/push` envelope + per-op `.strict()` Zod with session-sourced `cashierUserId`. **Issues.** L-3 login timing-attack window. No CSRF tokens — acceptable given SameSite=Strict + single origin, but worth documenting as a policy in the runbook.

**Recommendations.** Equalise login timing (L-3). Add a runbook note that SameSite=Strict is the CSRF defence of record and explain why CSRF tokens are intentionally absent.

### 4) Performance & Scalability

**Issues.** H-5 `/sync/pull` cursor=0 still loads every row. M-6 five other list endpoints unpaginated.

### 5) Reliability & Stability

**Issues.** H-1 oversold-void status bug. H-2 settled-event lock missing. M-1 no graceful cron shutdown. M-5 cursor-advances-on-partial-failure. M-7 orphan-lock gap. M-8 POS pay-failure ambiguity.
